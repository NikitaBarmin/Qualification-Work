4.1. Стек технологий
Frontend:
•	TypeScript
•	React + Vite
•	Redux Toolkit + RTK Query
•	FSD-архитектура (Feature-Sliced Design)
•	Ant Design (основная UI-библиотека)
•	SCSS Modules
•	Recharts (графики)
•	React Hook Form + Zod (валидация)
•	Axios (при необходимости)
Backend:
•	Node.js + Express
•	SQLite (на старте) → возможно PostgreSQL позже
•	Сырой SQL (или Prisma ORM — обсудим)
•	JWT + httpOnly cookies (рекомендуется)
4.2. Архитектура проекта
•	Frontend — строго по FSD
•	Backend — Layered Architecture — самый распространённый архитектурный паттерн в бэкенде. Идея элементарна: приложение разбивается на горизонтальные слои, каждый отвечает за свою часть работы. Слои общаются только сверху вниз — верхний вызывает нижний, но никогда наоборот. 🧱 Четыре классических слоя
🌐 Presentation (Controller) — принимает HTTP-запросы, извлекает данные из req, вызывает Service, формирует res. Это максимально тонкий слой — ноль бизнес-логики. Его задача: перевести HTTP-мир в мир чистых данных и обратно.
⚙️ Business Logic (Service) — сердце приложения. Вся бизнес-логика, валидация, оркестрация. Ключевое: Service не знает о HTTP, Express, gRPC, очередях. Он получает DTO и возвращает результат. Откуда пришёл запрос — ему неважно.
🗄️ Data Access (Repository) — работа с хранилищем. SQL-запросы, ORM-вызовы, кэш. Возвращает доменные объекты и прячет за собой детали конкретной БД. Завтра ты заменишь PostgreSQL на MongoDB — и изменится только этот слой.
💾 Database — конкретная технология хранения (PostgreSQL, MongoDB, Redis, файлы). Repository абстрагирует этот слой от остального приложения.
4.3. Правила генерации кода AI
Перед генерацией кода — прочитать актуальные docs/ (особенно product-overview и technical-spec)
Генерировать не более 200 строк за раз, спрашивать подтверждение
Добавлять комментарии только в сложных местах
Все новые экспортируемые функции и компоненты покрывать JSDoc
4.4. Definition of Done (DoD)
•	Код соответствует FSD
•	Типизация TypeScript (strict mode)
•	Компоненты переиспользуемые
•	RTK Query используется для всех запросов к серверу
•	Есть обработка ошибок и loading-состояния
•	Код прошёл код-ревью (или AI self-review)
2. Frontend (FSD)
Слои: app → pages → widgets → features → entities → shared
Импорты только сверху вниз (нельзя из shared импортировать features)
Внутри entities каждая сущность имеет свою папку: entities/dataset/model, entities/dataset/ui
3. Компоненты
Компонент — функция с большой буквы
Собственные стили — SCSS Module, имя ComponentName.module.scss
Пропсы типизировать интерфейсом IComponentNameProps
Если компонент использует данные из store, то подключение через useSelector внутри компонента
4. Состояние и запросы
Глобальное состояние только для данных, которые нужны в разных частях приложения (пользователь, датасеты, анализ)
Локальное состояние формы — useState или React Hook Form
Все API-запросы — через RTK Query (createApi). Ручной fetch/axios запрещён
Каждый запрос имеет transformResponse для типизации ответа
5. Обработка ошибок
В async thunk / query — try/catch с выбросом ошибки через rejectWithValue
На UI — показывать message.error из Ant Design
Пустые состояния (no data) — отдельный компонент с кнопкой действия
