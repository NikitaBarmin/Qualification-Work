DuckDB (OLAP — аналитика): Она создана для быстрой обработки больших таблиц и аналитических запросов. Ты будешь загружать в нее те самые «шаблонные таблички» пользователей для расчета метрик.
SQLite (OLTP — транзакции): Она нужна для хранения «состояния» твоего приложения: данные пользователей, настройки их профилей, логи посещений, права доступа в админке.
1. Общая архитектурная концепция
Твоя база данных — это не одна куча таблиц, а две независимые системы, связанные через Node.js:
SQLite (Постоянная): Работает по принципу 3NF (Третья нормальная форма). Здесь данные максимально разделены, чтобы избежать дублирования (например, почта пользователя хранится только в одном месте).
DuckDB (Временная): Работает по принципу Flat Table (Плоская таблица). Здесь нормализация не нужна — наоборот, чем меньше связей и чем «шире» таблица, тем быстрее она считается.

2. Сущности SQLite (Хранилище состояния)
В SQLite у нас будет 3 основные таблицы. Это «скелет» твоего приложения.
Таблица 1: users (Авторизация и профиль)
Хранит данные тех, кто пользуется сервисом.
id (UUID): Уникальный ключ.
email (String): Логин.
password_hash (String): Зашифрованный пароль.
business_type (Enum): Выбор из списка (SaaS, E-com, Локальный сервис). Это подсказка для ИИ.
created_at (Timestamp): Когда пришел юзер.
Таблица 2: datasets (Метаданные файлов)
Хранит информацию о загруженных файлах и том, как их интерпретировать.
id (UUID): Уникальный ключ.
user_id (FK -> users.id): Чей это файл.
filename (String): Красивое имя для UI.
file_path (String): Где лежит файл на диске сервера.
mapping_config (JSON): Самое важное. Здесь хранится схема сопоставления.
Пример: {"sys_revenue": "Выручка (руб)", "sys_date": "Дата транзакции"}.
created_at (Timestamp).
Таблица 3: analyses (Снимки/Snapshots результатов)
Здесь лежат «замороженные» результаты анализов. Это позволяет реализовать историю обновлений.
id (UUID): Уникальный ключ.
dataset_id (FK -> datasets.id): На основе какого файла сделан расчет.
user_id (FK -> users.id): Для быстрого поиска всех отчетов юзера.
kpi_metrics (JSON): Объект с 8 метриками (ROI, CAC и т.д.).
charts_data (JSON): Массивы для Recharts (динамика по времени).
swot_results (JSON): Текст SWOT-анализа от ИИ.
ai_recommendations (JSON): Список советов.
created_at (Timestamp).

3. Сущность DuckDB (Аналитическое ядро)
В DuckDB нет постоянных таблиц. В момент анализа Node.js создает одну временную таблицу:
Таблица: analytical_workspace
Структура: Это зеркало твоего CSV-файла, но с нормализованными именами колонок.
Процесс: Если у пользователя в Excel колонка называлась «Money», DuckDB при загрузке переименует её в revenue (на основе данных из datasets.mapping_config).
Почему так: Твои SQL-запросы для расчета ROI всегда будут искать колонку revenue. Маппинг позволяет коду быть универсальным для любого файла.

4. Как они связаны? (Workflow взаимодействия)
Схема взаимодействия «Бэк + SQLite + DuckDB» выглядит так:
Шаг «Маппинг»: Юзер загружает файл. Бэк читает заголовки, юзер связывает их. Бэк пишет в SQLite (datasets.mapping_config).
Шаг «Загрузка в DuckDB»: Бэк создает в памяти DuckDB таблицу и вставляет туда данные из файла, переименовывая колонки прямо в процессе INSERT или CREATE TABLE.
Шаг «Агрегация»: Бэк отправляет в DuckDB SQL-запросы:
SELECT AVG(order_value) FROM analytical_workspace — и получает цифру для SQLite.
Шаг «Финализация»: Бэк берет ответ от DuckDB и ответ от AI, упаковывает их в JSON и сохраняет как новую запись в SQLite (analyses).

1. Фаза инициации: Работа с SQLite (OLTP)
Регистрация метаданных: Бэкенд сохраняет запись в SQLite в таблицу Datasets. Мы фиксируем, кто загрузил файл, его оригинальное имя и, самое главное, сохраняем JSON-объект маппинга.
Создание черновика анализа: В таблице Analyses создается строка со статусом processing. Это нужно, чтобы если сервер упадет, пользователь видел: «Анализ в процессе, но что-то пошло не так».
2. Фаза подготовки (ETL-пайплайн)
Прежде чем считать метрики, данные нужно «причесать».
Чтение и трансформация: Node.js считывает файл (например, через csv-parser или xlsx). На лету он меняет названия колонок пользователя на системные (например, «Доход» становится revenue) согласно сохраненному маппингу.
Очистка: Бэкенд программно удаляет пустые строки, исправляет форматы дат и убирает явные дубликаты (возможно дубликаты будут убираться в DuckDB).
Сохранение «чистого» следа: Очищенные данные сохраняются в отдельный временный CSV-файл или передаются в буфер памяти для DuckDB.
3. Фаза вычислений: Работа с DuckDB (OLAP)
Это самый технологичный этап. Здесь Node.js «будит» аналитический движок.
Инициализация: Бэкенд создает подключение к DuckDB в режиме :memory:. Это гарантирует, что база не будет занимать место на диске и исчезнет сразу после расчетов.
Загрузка данных (Ingestion): Бэкенд отправляет SQL-команду в DuckDB: CREATE TABLE raw_data AS SELECT * FROM read_csv_auto('path/to/clean.csv'). DuckDB мгновенно поглощает файл и структурирует его.
Агрегация (Расчет 8 метрик):
Бэкенд отправляет серию тяжелых SQL-запросов. Например, для расчета ROI: SELECT (SUM(revenue) - SUM(costs)) / SUM(costs) * 100 FROM raw_data.
Для графиков: SELECT date, SUM(revenue) FROM raw_data GROUP BY date ORDER BY date.
Извлечение результатов: DuckDB возвращает результаты в Node.js в виде стандартных массивов и объектов JavaScript.
Завершение: Бэкенд закрывает соединение с DuckDB. Память очищается.
4. Фаза обогащения (AI) и финализации
Теперь у нас есть цифры, но нет «мозгов» — рекомендаций.
Запрос к AI: Бэкенд берет цифры из DuckDB (например: «ROI: 80%, Конверсия: 1.2%») и отправляет их в API нейросети. Он просит: «На основе этих данных сделай SWOT и дай 3 совета».
Формирование Snapshot (Снимка): Бэкенд собирает итоговый «пакет» данных:
Метрики из DuckDB.
Массивы для графиков из DuckDB.
SWOT-анализ и текст рекомендаций от ИИ.
5. Фаза записи: Снова SQLite (Snapshot)
Чтобы пользователь мог увидеть этот отчет через неделю или сравнить его с новым, мы должны «зацементировать» результат.
Обновление статуса: Бэкенд находит ту самую строку в SQLite, которую создал на первом этапе.
Запись данных: Он записывает весь собранный пакет (метрики, графики, советы) в соответствующие JSON-колонки таблицы Analyses. Статус меняется на completed.
Ответ клиенту: Только теперь React получает финальный ответ и отображает дашборд.
Тонкости и нюансы взаимодействия:
Асинхронность: Взаимодействие с DuckDB в Node.js часто требует использования Promises, так как стандартная библиотека работает на колбэках. Это нужно учитывать, чтобы бэкенд не «зависал».
Целостность данных: Если ИИ-сервис выдаст ошибку, бэкенд должен поймать её и всё равно сохранить хотя бы графики и метрики из DuckDB в SQLite, пометив отчет как «Частично успешный».
